# GEO 前端 V2 AI 分工表

## 1. 分工原则

- `GPT-5.4` 作为主模型，负责工程落地、接口对接、状态管理、代码质量和最终收口
- `Gemini 3.1 Pro` 作为设计副模型，负责视觉方案、页面变体、图表表达和交互审美 review
- 所有最终进入仓库的代码，以 `GPT-5.4` 收口版本为准
- `Gemini 3.1 Pro` 输出优先作为设计参考、原型建议、视觉优化建议，不直接作为最终 merge 依据

## 2. 总体分工

| 维度 | GPT-5.4 | Gemini 3.1 Pro |
|---|---|---|
| 工程骨架 | 主负责 | 参与建议 |
| API 对接 | 主负责 | 不主导 |
| 类型定义 | 主负责 | 不主导 |
| 状态管理 | 主负责 | 不主导 |
| 页面视觉 | 配合落地 | 主负责 |
| 图表表达 | 配合实现 | 主负责 |
| 可视化创意 | 配合实现 | 主负责 |
| 代码重构 | 主负责 | 辅助 review |
| 代码 review | 主负责 | 辅助 review |
| 最终验收 | 主负责 | 辅助审美验收 |

## 3. 按阶段分工

| 阶段 | 目标 | GPT-5.4 负责 | Gemini 3.1 Pro 负责 | 完成标准 |
|---|---|---|---|---|
| Phase 0 | 架构定稿 | 定目录、路由、数据层边界、模块拆分 | 提出版式建议、视觉风险提醒 | 技术栈、目录、模块边界冻结 |
| Phase 1 | 骨架搭建 | 初始化项目、接 Router/Pinia/Vue Query/Axios、基础布局 | 侧栏、卡片、暗黑风样式建议 | 项目可运行，主布局可浏览 |
| Phase 2 | 基础组件库 | `StatCard`、`StatusLabel`、`IndustrialTable`、`DetailPanel` 等组件工程实现 | 组件视觉细节、间距、状态色建议 | 组件可复用，样式统一 |
| Phase 3 | 核心页面一期 | 总览、运行、内容、发布、系统状态的真实页面实现 | 页面层级、图表样式、空态错误态优化 | 5 个核心页面接通真实 API |
| Phase 4 | 预留模块 | 关键词、能力库、图谱的占位页和扩展位 | 对应页面原型、视觉草案 | 路由和模块占位完整 |
| Phase 5 | 质量收口 | 重构、类型补全、分包优化、错误边界、构建优化 | 审美 review、交互一致性 review | 可以作为正式 V2 起点继续开发 |

## 5. 按模块分工

| 模块 | GPT-5.4 | Gemini 3.1 Pro |
|---|---|---|

| `api` | 建 Axios 实例、拦截器、响应解包 | 不主导 |

| `composables` | 定义 Vue Query hooks | 不主导 |
| `stores` | 只保留 UI 状态和本地筛选状态 | 不主导 |
| `layouts` | 实现主布局与导航 | 辅助优化视觉层次 |

| `charts` | ECharts 配置落地 | 图表表达优化 |

## 6. 明确职责边界

### 6.1 GPT-5.4 不该让给 Gemini 的内容

- API 对接和错误处理
- Vue Query key 设计
- Pinia 与 Vue Query 边界
- 类型定义和数据解包
- 路由和模块边界
- 重构和性能优化
- 最终 merge 前代码收口

### 6.2 Gemini 3.1 Pro 最适合承担的内容

- 暗黑工业风视觉 refinement
- 卡片、表格、抽屉、筛选区布局推敲
- ECharts 配色、层级、信息密度建议
- 知识图谱、网络图、复杂可视化的原型探索
- 空态、错误态、低数据量场景的观感优化

## 7. 推荐协作流程

### 流程 A：页面开发

1. `GPT-5.4` 先定义页面数据源、接口、状态边界
2. `Gemini 3.1 Pro` 输出该页面的视觉建议或变体
3. `GPT-5.4` 把视觉方案落成真实 Vue 代码
4. `Gemini 3.1 Pro` 做一次审美和交互 review
5. `GPT-5.4` 根据 review 收口并提交

### 流程 B：复杂图表或知识图谱

1. `Gemini 3.1 Pro` 先出结构草案和视觉交互建议
2. `GPT-5.4` 选定技术实现路线，例如 ECharts / D3 / G6
3. `GPT-5.4` 做性能和数据接入
4. `Gemini 3.1 Pro` 对表现层再做微调建议

## 8. 当前最推荐的执行顺序

| 顺序 | 任务 | 主模型 | 副模型 |
|---|---|---|---|
| 1 | 补齐 `src/api`、`src/types`、`src/composables` | GPT-5.4 | - |
| 2 | 补 `MainLayout` 的移动端导航和主题细节 | GPT-5.4 | Gemini 3.1 Pro |
| 3 | 建基础组件库 | GPT-5.4 | Gemini 3.1 Pro |
| 4 | 先完成总览页 | GPT-5.4 | Gemini 3.1 Pro |
| 5 | 再完成运行中心 | GPT-5.4 | Gemini 3.1 Pro |
| 6 | 再完成内容中心和发布中心 | GPT-5.4 | Gemini 3.1 Pro |
| 7 | 最后补系统状态、占位页和分包优化 | GPT-5.4 | Gemini 3.1 Pro |

## 9. 验收标准

满足以下条件，前端 V2 才能算进入“可持续开发”状态：

- 路由完整
- API 层完整
- Vue Query 数据层完整
- 基础组件库可复用
- 5 个核心页面可接真实接口
- `keywords / capabilities / graph` 有占位页和扩展位
- 构建可通过
- 首屏布局和主流程在桌面端稳定可用

## 10. 一句话结论

`GPT-5.4` 负责“把系统做对、做稳、做可维护”，`Gemini 3.1 Pro` 负责“把页面做得更好看、更有设计感、更有图表表现力”。这套分工最适合 GEO 控制台 V2 这种既要工程化、又要工业审美的项目。
